# Tortuga OS — Story Format (Agentic-Ready)

> Canonical schema for User Stories produced in F1_SALES and consumed by every subsequent phase.
> Companion to [DOMAIN.md](./DOMAIN.md) and [PHASES.md](./PHASES.md).

---

## Why a strict format

A Story must be executable by an agent (human or AI) without re-asking the orchestrator. If a Dev reads the Story and still has to ping Sales to figure out what to build, the Story failed at F1 and the cost of that ping is rework against F1_SALES.

The format below is mandatory. F1 cannot close until every Story validates against it.

---

## On-disk layout

```
05-build/<STORY_ID>/
  story.md        ← human-readable brief (frontmatter YAML + markdown body)
  story.yaml      ← machine-readable companion (generated from story.md, kept in sync)
  tasks/
    <TASK_ID>/
      ...
```

`story.md` is the single source of truth. `story.yaml` is generated by the orchestrator from the frontmatter for programmatic consumption (gate runners, cost calculator, agents).

Once F1_SALES is approved, `story.md` is **immutable**. Any change requires either:
- A new Quote version (if scope changes), or
- A ReworkTicket (if the story was wrong from the start).

## Required schema

### Frontmatter (YAML)

```yaml
---
id: GASTUU-014                       # <PROJECT_CODE>-<NNN>
title: "Implementar login con email/password"
quote_version: 1                     # quote it belongs to
priority: 2                          # 1 = highest
estimated_hours:
  designer: 4
  tech_lead: 2
  dev: 12
  qa: 3
  devops: 1
owner_role: dev                      # primary executor in F5
status: pending                      # pending | in_progress | qa | approved | rejected
inputs:
  design:
    figma_file_key: <FIGMA_FILE_KEY>
    figma_node_ids:
      - "12:3402"
      - "12:3501"
  api_contract: 04-architecture/contracts/auth.openapi.yaml
  brand_tokens: tortuga-projects/<PROJECT_CODE>/03-design/figma-links.md
  references: []                     # optional links
outputs:
  files:
    - lib/features/auth/presentation/login_page.dart
    - lib/features/auth/application/login_bloc.dart
    - lib/features/auth/data/auth_repository.dart
  endpoints: []                      # for api stories
  evidence:
    - type: video
      description: "Flujo completo de login exitoso y con credenciales inválidas"
    - type: screencap
      description: "Estados: idle, loading, error, success"
verification:
  gates: [G1, G2, G3, G4, G5, G6, G7]
  manual_checks:
    - "Login con credenciales válidas redirige a /home"
    - "Login con credenciales inválidas muestra error inline, no toast genérico"
    - "Botón 'Olvidé mi clave' navega a /forgot-password"
acceptance_criteria:
  - id: AC1
    given: "Usuario en pantalla de login"
    when: "Ingresa email válido y password correcto y presiona 'Ingresar'"
    then: "Es redirigido a /home en menos de 2 segundos"
  - id: AC2
    given: "Usuario en pantalla de login"
    when: "Ingresa credenciales inválidas"
    then: "Ve un mensaje de error inline (no toast) que dice exactamente 'Email o clave incorrectos'"
  - id: AC3
    given: "Usuario en pantalla de login con sesión previa"
    when: "Reabre la app"
    then: "Salta el login y va directo a /home si el token sigue válido"
out_of_scope:
  - "Login con Google/Apple (otra historia)"
  - "Recuperación de clave (otra historia)"
  - "Registro de cuenta nueva (otra historia)"
---
```

### Body (Markdown)

```markdown
## Goal

[2-4 sentences explaining the outcome from the user's perspective. No
implementation detail. No screen names. Just what the user achieves and why
it matters.]

## Context

[Any constraints that the YAML doesn't capture: regulatory, performance,
security, existing-system quirks. Optional. If empty, write "Ninguno".]

## Notes for the executing agent

[Anything an agent needs to know that doesn't fit the acceptance criteria
but might trip them up: error message copy that came from legal, a known
quirk in the auth backend, a previous failed attempt in another project
worth not repeating. Optional.]
```

## Validation rules (enforced by orchestrator)

A Story is **invalid** (F1 cannot close) if any of these fail:

| Rule | Reason |
|---|---|
| `id` matches `^<PROJECT_CODE>-\d{3,}$` | Stable identifier. |
| `title` is ≤ 80 chars and starts with an imperative verb | Force clarity. |
| `estimated_hours` has at least one role with `> 0` | Cost has to land somewhere. |
| `inputs.design.figma_file_key` and `figma_node_ids` present if `owner_role` is `designer` or `dev` and surface is UI | Design A is canon (see memory). |
| `outputs.files` is non-empty for `dev`-owned stories | Real work must produce real artifacts. G6 enforces this. |
| `outputs.evidence` lists at least one item | Every Story produces evidence. |
| `acceptance_criteria` has ≥ 1 entry | Without AC, the story is unfalsifiable. |
| Every `acceptance_criteria` entry has `given`/`when`/`then` non-empty | Force testable formulation. |
| `verification.gates` is a non-empty subset of `[G1..G7]` | Gates per task type are defined in PHASES.md gate matrix. |
| `out_of_scope` exists (can be empty array but must be declared) | Force the author to think about boundaries. |

The orchestrator runs these checks; F1_SALES cannot transition to `approved` until every Story passes.

---

## Anti-patterns the format prevents

| Anti-pattern | How the schema blocks it |
|---|---|
| "Build the login screen" | Missing `acceptance_criteria` and `inputs.design.figma_node_ids` → invalid. |
| Scope creep mid-build | Story is immutable after F1 approval; any change is a ReworkTicket or a Quote v+1. |
| "It works on my machine" | `verification.gates` is mandatory; `outputs.evidence` is mandatory. |
| "QA didn't have a spec to check against" | `acceptance_criteria` is the spec. |
| Designer + Dev disagreement on what the screen should look like | `inputs.design.figma_node_ids` is the single visual truth. |
| Estimating one bucket of hours for the whole team | `estimated_hours` is per-role, so cost imputation works correctly. |

## Example: minimal valid Story

```yaml
---
id: CAM-007
title: "Mostrar lista de cámaras del usuario"
quote_version: 1
priority: 3
estimated_hours:
  dev: 6
  qa: 1
owner_role: dev
status: pending
inputs:
  design:
    figma_file_key: <FIGMA_FILE_KEY>
    figma_node_ids: ["18:2201"]
  api_contract: 04-architecture/contracts/cameras.openapi.yaml
outputs:
  files:
    - lib/features/cameras/presentation/cameras_list_page.dart
    - lib/features/cameras/application/cameras_bloc.dart
  evidence:
    - type: video
      description: "Lista vacía, lista con 3 cámaras, error de red"
verification:
  gates: [G1, G2, G3, G4, G5, G6, G7]
  manual_checks: []
acceptance_criteria:
  - id: AC1
    given: "Usuario autenticado con 0 cámaras"
    when: "Abre la pantalla de cámaras"
    then: "Ve estado vacío con CTA 'Agregar cámara'"
  - id: AC2
    given: "Usuario autenticado con N>0 cámaras"
    when: "Abre la pantalla de cámaras"
    then: "Ve la lista ordenada por last_seen DESC con thumbnail, nombre, estado online/offline"
out_of_scope:
  - "Agregar/editar/eliminar cámara"
  - "Stream en vivo de la cámara"
---

## Goal

El usuario puede ver de un vistazo cuáles de sus cámaras están en línea y
cuál fue su última actividad, para decidir si necesita revisar algo.

## Context

Ninguno.

## Notes for the executing agent

El endpoint ya está disponible en QA, ver `cameras.openapi.yaml`. El campo
`last_seen` viene en ISO-8601; mostrar como "hace X minutos" usando el
helper ya existente en `lib/shared/time_ago.dart`.
```

This Story is executable. A Dev can read it and start implementing without asking anyone anything.
